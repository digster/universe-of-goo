# 2026-04-19 — Self-contained offline docs (file:// works)

## What shipped

The interactive doc demos now run correctly when opened via `file://`
after `npm run build`. The README's promise ("double-click
`dist/docs/index.html`, no server needed") was broken: the built pages
still used ES-module `<script type="module">` with
`<link rel="modulepreload">` pointing to separate asset chunks. Browsers
refuse cross-chunk `import` statements when the origin is `null`
(file://), so every embedded demo canvas was blank.

The build now emits each of the 8 docs HTML entries
(`dist/docs/index.html` + `dist/docs/pages/01..07-*.html`) as **one
fully self-contained file** with all JS and CSS inlined. Game and
sandbox are unchanged — they still use a normal chunked Vite build and
require HTTP (they `fetch()` level JSON at runtime).

## Why this way

`vite-plugin-singlefile` does exactly what's needed: enables
`inlineDynamicImports`, disables CSS code splitting, inlines
modulepreload chunks, and rewrites the emitted HTML to contain its
dependency graph. Rebuilding that by hand is ~200 lines of
HTML-rewriting + Rollup-chunk-relinking that would rot quickly.

Two constraints forced the shape of the fix:

1. **`inlineDynamicImports: true` is incompatible with multi-input
   Rollup builds.** So the docs entries can't stay alongside game +
   sandbox in one `vite.config.js` input map. They had to move.
2. **Even a dedicated multi-entry `vite.docs.config.js` fails** for
   the same reason — Rollup rejects it at validation. Solution: loop
   over the 8 entries one-by-one, running a single-input Vite build
   each time. That's `scripts/build-docs.js`.

Each per-entry build writes into the same `dist/` with
`emptyOutDir: false`, so the first pass (game + sandbox) is preserved.
Total overhead: ~8 small Vite builds, each ~60ms.

## Observations worth remembering

- **`vite-plugin-singlefile` + `inlineDynamicImports` + multi-input is
  a triangle you cannot close.** Rollup explicitly errors with
  `"Invalid value for option 'output.inlineDynamicImports' - multiple
  inputs are not supported when 'output.inlineDynamicImports' is true."`
  The only principled fix is to loop: one Vite build per entry. Do not
  try to monkey-patch the plugin or disable its recommended config —
  the inlining depends on all those options being set together.
- **`base: './'` does NOT make a build file://-safe.** It only
  controls *path style* of the emitted `<script src>` / `<link href>`
  URLs (relative vs absolute). The CORS `null`-origin block on ES
  module imports is a completely separate protection layer that
  relative paths do nothing about. You need the JS itself to have no
  inter-file `import` statements — i.e. inlining.
- **Noisy Vite modulepreload polyfill.** The inlined JS contains
  `document.querySelectorAll('link[rel="modulepreload"]')` as part of
  Vite's runtime preload polyfill. A naive `grep 'modulepreload'` over
  `dist/docs/` returns 7 matches — but these are inside the inlined
  script body, not actual `<link>` tags. When auditing, grep for the
  exact HTML tag (`'<link rel="modulepreload"'`) to avoid false
  positives.
- **Pre-existing broken links flagged, not fixed.** Each doc page
  includes anchors like `<a href="../../src/physics/integrator.js">`.
  These 404 under `dist/` (no `src/` tree is copied). This has always
  been broken offline and is orthogonal to the demo-CORS fix. Fixing
  it would mean either (a) rewriting anchors to GitHub URLs at build
  time, or (b) copying `src/` into `dist/`. Left as a future-work
  item.
- **Duplication cost.** Each of the 8 docs HTMLs inlines its own copy
  of the harness + shared physics modules (~5–10 KB × 8). Acceptable
  for a pedagogical site; if the page count grows, consider
  deduplicating via a shared external script (but only on HTTP — it
  would break file://).

## Commit message suggestion

```
fix(docs): make built docs pages runnable from file:// (self-contained HTMLs)

The README and vite.config.js comments both promised that after
`npm run build`, double-clicking `dist/docs/index.html` or any
`dist/docs/pages/*.html` would run the embedded demos with no server.
That was broken: the built pages still used ES-module `<script type=
"module">` with cross-chunk `import` statements, which browsers refuse
over a file:// (null) origin. Every demo canvas stayed blank with
"Origin null is not allowed by Access-Control-Allow-Origin" errors.

Fix: split the build. The main `vite build` now only emits the game +
sandbox entries (those fetch level JSON and need HTTP anyway). A new
`scripts/build-docs.js` loops over the 8 docs HTML entries, running a
single-input Vite build per entry with `vite-plugin-singlefile`, which
inlines all JS + CSS into the HTML. Each `dist/docs/**/*.html` is now
one fully self-contained file — no <script src=>, no <link href=>, no
modulepreload — runnable from disk.

A per-entry loop is required because `vite-plugin-singlefile` sets
`output.inlineDynamicImports: true`, which Rollup rejects with multiple
inputs. `emptyOutDir: false` on the docs passes preserves the game +
sandbox output from the first pass.

Adds a file:// smoke test block to e2e/docs.spec.js that loads
dist/docs/pages/01-particle.html and 04-cloth.html directly from disk
via pathToFileURL and asserts the canvas renders + no console errors.
README.md and ARCHITECTURE.md updated with the new build model.

npm test: 41 tests green. npm run e2e: 23 tests green (21 + 2 new).
```
