# 2026-04-18 — Drag-attachment preview

## What shipped

Added an "honest preview" of the strands that would form when a free walker
is released onto the structure. While the player is mid-drag in game mode,
faded dashed goo-yellow lines now radiate from the walker to each ball the
release would connect to — identical to what `attachFreeBall()` will actually
do on `pickEnd`.

**Scope:** game mode only (`src/main.js`). Sandbox interactions were
explicitly out of scope.

## Why this way

`attachFreeBall()` in `src/game/strandRules.js` was already factored as a
pure query (`pickAttachmentCandidates`) + a side-effecting writer. We call
the pure query every frame during drag and feed the results into the
existing `scene.drags[]` pipeline the renderer already consumes. No new draw
layer, no new helper, no physics involvement.

The preview uses `radius: 160` + `maxCount: type.maxStrands` — the SAME
arguments `attachFreeBall` uses internally at
`src/game/strandRules.js:92`. If those ever drift, `tests/preview.test.js`
fails (it compares the preview set to the actual strand set produced on
release).

## Files touched

- `src/main.js` — import `pickAttachmentCandidates`; add preview entries to
  the per-frame `drags` array when the dragged ball is a free walker.
- `src/render/renderer.js` — handle a `preview: true` flag on drag entries.
  Renders `hsl(45, 70%, 55%)` at `globalAlpha = 0.45`, dashed `[6, 4]`, width
  3.  Cursor rubber-band line unchanged.
- `tests/preview.test.js` (new) — 5 vitest unit tests that lock in the
  "preview = reality" invariant: preview candidates equal the actual
  attachment targets for in-radius / out-of-radius / line-of-sight /
  saturated / nearest-K scenarios.
- `e2e/preview.spec.js` (new) — 2 Playwright tests using real `page.mouse`
  that sample canvas pixels for goo-yellow along the expected preview strip
  mid-drag, and confirm nothing paints when the drop is out of range.

## Results

- `npm test` → 41 vitest tests green (36 pre-existing + 5 new).
- `npx playwright test` → 21 e2e tests green (19 pre-existing + 2 new).

## Observations worth remembering

- **`setPointerCapture` + synthetic `PointerEvent` don't mix.**  Manual
  `dispatchEvent` of synthetic pointer events fails silently because
  `canvas.setPointerCapture(e.pointerId)` in
  `src/input/pointer.js:62` throws when `e.pointerId` doesn't correspond to
  a real captured pointer, short-circuiting the rest of `_down()`. Real
  Playwright `page.mouse.down/move/up` sidesteps this entirely. Lesson: for
  canvas drag tests, always use the real driver, not `dispatchEvent`.
- **Level-01's `basic` balls are over-saturated.** The level JSON
  (`public/levels/01-first-tower.json`) wires 3 strands onto each `basic`
  ball, whose `maxStrands = 2`. So every basic ball is beyond its budget and
  filtered out by `pickAttachmentCandidates`. The only attachable ball is
  the pinned `fixed` anchor (`maxStrands = 4`, using 2). This is now
  surfaced honestly by the preview — if a player drops near a basic ball
  they see no preview and wonder why. Worth revisiting level authoring:
  either raise `basic.maxStrands` or thin out the initial strand list.

## Commit message suggestion

```
feat(game): preview attachment strands while dragging free walkers

While a free ball is held mid-drag, draw faded dashed goo-yellow lines from
the walker to each ball that attachFreeBall() would connect to on release.
Reuses pickAttachmentCandidates() with identical params (radius 160,
maxCount = type.maxStrands) so the preview is always honest; unit tests in
tests/preview.test.js compare the preview set against the actual strand set
created by attachFreeBall() to prevent drift.

Game mode only (src/main.js); sandbox interactions unchanged. Playwright
e2e spec samples canvas pixels to verify the preview paints mid-drag and
doesn't paint when the drop is out of the attachment radius.
```

# 2026-04-18 — Sandbox mode aligned with game-mode dynamics

## What shipped

The sandbox now uses the same gestures and the same attachment rules as
the main game. The bespoke shift-drag-to-link gesture is gone. Spawned
balls behave like game-mode walkers (free walkers fall and only attach on
release; `fixed` balls remain anchors). While a held ball is mid-drag, the
same dashed goo-yellow attachment preview that game mode uses now appears
in the sandbox too.

## Why this way

Shipping "sandbox = game mode minus the goal" lets the sandbox demonstrate
the exact same `strandRules.js` logic the docs already explain, instead of
diverging into its own gesture vocabulary. The pickup/release pipeline now
calls `pin/unpin/warpTo` + `attachFreeBall` directly (mirroring `src/main.js`)
rather than going through controller wrappers, so any future tweak to game
mode's drag/attach pipeline ports to the sandbox by symmetry.

`makeGooBall()` already had the right default (`attached: opts.attached ??
pinned`) — `fixed` balls auto-mark attached=true, all others default to
false. The old `spawn()` was *fighting* that default by force-setting
`attached = true`. Removing the override cleared the path.

## Files touched

- `src/sandbox.js` — replaced the three-kind drag dispatch (moveBall /
  linkBall / spawn) with the two-kind game-mode-aligned version
  (moveBall / spawn). pickStart pins the held ball with `pin()`, drag
  warps it with `warpTo()`, pickEnd unpins and runs `attachFreeBall()`
  for free walkers. Per-frame drag-rendering block emits the same
  `{from, to, attachable, preview: true}` shape `src/main.js` uses, so
  the renderer paints the same dashed candidate strands.
- `src/game/sandboxController.js`:
  - `spawn()` simplified — drops the `{ attached: false }` override and
    the `ball.attached = true` post-step, relying on `makeGooBall`'s
    `attached: opts.attached ?? pinned` default. `fixed` → anchor;
    everything else → free walker.
  - `linkBetween()` now sets `attached = true` on both balls, so the
    seed scene (and snapshot restore) leave their balls in a state
    where they're valid attachment candidates for later drags.
  - Removed the now-unused `grab()` / `release()` / `drag()` helpers
    and the `warpTo` import.
  - Updated the top docstring.
- `sandbox.html` — help footer no longer mentions shift-drag; describes
  the auto-attach + dashed preview behavior.
- `README.md` — sandbox controls section rewritten to match.

## Results

- `npm test` → 41 vitest tests green (no regressions).
- `npm run e2e` → 21 Playwright tests green (no regressions). The
  existing `e2e/sandbox.spec.js` only exercises toolbar + spawn so no
  test changes were required; the existing `e2e/preview.spec.js` already
  guards the preview-vs-attach parity invariant the sandbox now
  inherits.

## Observations worth remembering

- **`makeGooBall`'s `attached: opts.attached ?? pinned` default is the
  right contract.** Anywhere downstream that wants game-mode semantics
  should *not* override it. The old sandbox code overriding it was a
  symptom of mistaken "sandbox-only" thinking — the game-mode default
  is what we wanted all along.
- **Pinned balls (fixed type or middle-click pinned) are skipped on
  pickup**, mirroring game mode's `if (hit.pinned) return;`. To move a
  pinned ball in the sandbox you middle-click first to unpin. This is
  intentional — letting users drag pinned balls would break the anchor
  semantics they were spawned to provide.
- **`linkBetween` is now an internal API.** It's still used by the
  starter seed and snapshot restore but no longer by any user gesture.
  Worth keeping in mind if a future contributor proposes "let's expose
  manual linking in the sandbox" — the architectural goal here is the
  opposite, sandbox = game without the goal.

## Commit message suggestion

```
feat(sandbox): align with game-mode dynamics; remove shift-drag linking

The sandbox no longer has its own gesture vocabulary. Picking up a ball
pins it to the cursor; on release, free walkers (zero strands) auto-
attach via the same strandRules.js the game uses, with the same dashed
goo-yellow attachment preview painted mid-drag. The shift-drag manual
linking gesture is removed entirely.

Spawn semantics now mirror the game: fixed balls are anchors
(attached=true, pinned), all other types spawn as free walkers
(attached=false) and fall under gravity until the player drags + drops
them onto the structure. Implemented by removing the explicit override
in SandboxController.spawn() and letting makeGooBall's
`attached: opts.attached ?? pinned` default do its job.

linkBetween() is retained as an internal API for the starter seed and
snapshot restore, but is no longer surfaced as a user gesture. The
unused grab/release/drag controller helpers are deleted.

Help text in sandbox.html and README.md updated. All 41 unit tests and
21 Playwright e2e tests pass.
```
